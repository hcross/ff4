# What "reverse-engineering a port" actually means here

## Disassembly vs. decompilation — a distinction worth being precise about

**Disassembly** turns raw machine code back into assembly text — a
mechanical, lossless translation (opcode bytes → their mnemonic form).
This project starts from an existing, hand-annotated disassembly of Final
Fantasy IV's ROM (`ff4-port/upstream/notes/ff4j-sfc.asm`) produced by the
community over years of manual analysis — labels, comments, and structure
added by humans reading the raw disassembly.

**Decompilation** would go further: automatically reconstructing
higher-level, structured C-like code from the assembly — the way tools
like Ghidra do for compiler-generated binaries, by recognizing patterns a
known compiler tends to produce. That doesn't apply here: FF4's assembly
was **hand-written** by human programmers in 1991, not generated by a
compiler with a recognizable fingerprint. There is no decompiler for
hand-written 65816 assembly, and this project doesn't attempt to build
one — see `ff4-port/ca65-bridge`'s own capability table, which honestly
records `has_decompile: no`.

So the actual task is: **read** the disassembly (assisted by tooling,
`ca65-bridge`), and **rewrite** each routine by hand (or with LLM
assistance) as an equivalent C function — "equivalent" being the operative
word, which is where the next section comes in.

## The core technique: shadow execution

How do you know a hand-rewritten C function actually behaves identically
to the original assembly, across every input the game might ever throw at
it — not just the one case you thought to test? You don't reason it out by
eye alone; you **run both versions from the same starting state and
compare the results, mechanically, for every routine you port.**

This project (and its direct methodological ancestor,
[`zelda3`](https://github.com/snesrev/zelda3) — a from-scratch C
reimplementation of *A Link to the Past* that plays the whole game
start-to-finish, built the exact same way) calls this **shadow execution**:
the original game keeps running inside a SNES emulator (this project uses
[LakeSnes](https://github.com/elzo-d/LakeSnes)) "in the shadow" of the new
C code, and after each unit of work the two are compared byte-for-byte —
CPU registers, memory, whatever the routine is expected to affect. Any
difference is caught immediately, at the exact point it happened, instead
of surfacing later as "something is subtly wrong three screens later" —
which is exactly the kind of bug this technique is built to catch early.

`zelda3`'s siblings [`smw`](https://github.com/snesrev/smw) (Super Mario
World) and [`sm`](https://github.com/snesrev/sm) (an early-stage Super
Metroid port — its own README describes it as still buggy), under the same
[snesrev](https://github.com/snesrev) organization, and the
disassembly-tooling project [`snes2asm`](https://github.com/nathancassano/snes2asm),
all reinforce the same lesson from independent SNES reverse-engineering
efforts: **treat the original ROM, running under emulation, as the source
of truth — never trust a rewrite that hasn't been mechanically compared
against it.**

## From one routine to a whole game: the maturity ladder

Because shadow-comparing an entire game's execution at once is a big,
hard-to-debug jump, this project breaks the proof into incremental stages
— a "maturity ladder" a routine climbs one rung at a time (the full,
authoritative definition is `AGENTS.md` §B.2; this is the same ladder in
plain language with a worked example):

1. **L0 — stub.** The routine has a slot in the dispatch table (see the
   [glossary](00-glossary.md)) but no real body yet — it's a placeholder.
2. **L1 — ported, unproven.** Someone (human or LLM) has written a C
   version. It compiles. Nobody has checked it against the original yet —
   it might be completely wrong.
3. **L2 — per-routine equivalence proven.** A **spike** (see the
   glossary) runs the C version and the original assembly (interpreted)
   from the same starting state, feeds them the same inputs, and confirms
   they produce the same outputs. This is shadow execution applied to
   exactly one routine, in isolation.
4. **L3 — validated in a real game scenario.** The routine is exercised
   as part of an actual gameplay moment (loaded from a real savestate),
   with shadow execution comparing the whole system's memory, not just
   the one routine's declared inputs/outputs — catching interactions the
   isolated spike couldn't see (a routine that reads memory it never
   declared, for instance).
5. **L4 — proven on the real device.** The same code, now compiled for
   the actual Game & Watch hardware, runs without crashing and produces
   the expected result on a real screen — the final check, because
   desktop and embedded hardware genuinely differ (see `AGENTS.md`'s note
   on the Game & Watch's much slower, in-order CPU).

**Worked example.** Say a routine computes battle damage. At L1, it's a
plausible-looking C function nobody has run yet. Getting to L2 means a
spike feeds it the same attacker/defender stats the original assembly
would see, from the same CPU/memory state, and confirms both produce the
identical damage number — for many randomized inputs, not just one. Only
after that passes does the routine get wired into a real battle scenario
(L3) and eventually flashed to the device (L4). A routine can look
completely correct by eye and still fail its spike — that's the entire
point of proving it mechanically rather than trusting a read-through.
