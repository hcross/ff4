# 65816 assembly, just enough to read this project's code

> This is not a general assembly-programming tutorial. It's the minimum
> vocabulary needed to read the disassembled Final Fantasy IV source and
> understand what the pitfalls documented across this project are actually
> about. For the full, precise reference, see
> [`04-further-reading.md`](04-further-reading.md).

## What assembly even is, briefly

A CPU only understands raw numeric instructions ("opcodes"). Assembly
language is a human-readable one-to-one text form of those instructions —
`LDA $05` means "load the accumulator with the value found at address
`$05`," and it corresponds to one specific opcode byte. There's no
abstraction here (no functions-with-types, no automatic memory
management) — every instruction does exactly one small, literal thing to
the CPU's registers or to memory.

## The registers — think of them as a handful of global variables

The 65816 CPU (used in the SNES; a 16-bit evolution of the 6502 used in
the NES) has a small, fixed set of registers:

- **A** — the accumulator. Most arithmetic and logic happens here.
- **X, Y** — index registers, mostly used to walk through arrays/tables.
- **D** — the **Direct Page** register. See the trap below — this one
  matters enormously.
- **DB** — the **Data Bank** register: supplies the bank (the high part of
  a 24-bit address) for a certain class of memory access.
- **S** — the stack pointer.
- **P** — the processor status (flags: zero, carry, negative, overflow,
  and two width flags described below).

A and X/Y can each be either **8-bit or 16-bit**, switched at runtime via
two flags in P (the `m` flag for A, the `x` flag for X/Y), using the
`REP`/`SEP` instructions. This matters because *the same instruction text*
means something different depending on the current width — `LDA $05` loads
one byte in 8-bit mode, two bytes in 16-bit mode. Reading disassembled code
correctly requires tracking these flags as you go; getting this wrong is a
common, subtle source of bugs (see Pitfalls 6-9 in
[`ff4-port/prompts/pitfalls.yaml`](https://github.com/hcross/ff4-port/blob/main/prompts/pitfalls.yaml)).

## §3 — The Direct Page trap (read this one twice)

A short instruction operand like `$05` (two hex digits) is a **Direct
Page** access: the CPU computes the real address as `D + $0005`, always in
bank `$00`. When `D` is `$0000` (its default), `$05` simply means
`$000005` — which, thanks to a memory mirror, is the same physical byte as
`$7E0005` in WRAM. So far, unsurprising.

The trap: **`D` is not always zero.** Game code frequently repoints `D` to
a different base — for example, this project's `AGENTS.md` documents that
FF4's field/NMI code runs with `D=$0600`, while its menu code runs with
`D=$0100`. The *exact same instruction text*, `LDA $05`, then reads a
*completely different* physical address depending on which routine you're
in and what it set `D` to beforehand. You cannot know what a short operand
points to without separately knowing the current value of `D` — and that
value is not written down next to the instruction, you have to trace it
from context (or, in this project, from the per-routine facts recorded in
`ff4-gnw/CONVENTIONS.md`).

This is called out because it is, empirically, the single most common
cause of bugs in this kind of port — see `AGENTS.md`'s "golden rule #1" and
the `DP_SENSITIVE` flag tracked throughout `DISPATCH_REGISTRY.md`.

`DB` (Data Bank) is a related but *different* register: it supplies the
bank for a longer, 4-digit operand (`$1234` rather than `$05`). Confusing
"which register supplies which kind of address" is its own separate
pitfall class — see Pitfall 14 in `pitfalls.yaml`.

## Subroutines, and a rule that must never be broken

`JSR`/`JSL` call a subroutine (same-bank / any-bank respectively) and push
a return address; `RTS`/`RTL` return by popping it. **A call and its
return must match** — a `JSR` must be matched by `RTS`, never `RTL`, and
vice versa for `JSL`/`RTL`, or the stack ends up corrupted and the program
counter jumps somewhere nonsensical. The same discipline applies to
`PHx`/`PLx` (push/pull) pairs: pull exactly as many bytes as you pushed, in
the same width, or the eventual return address is wrong.

## Reading further

This page intentionally covers only the two ideas (register width, the D/DB
trap) that recur constantly across this project's own pitfall list and
architecture notes. For the complete instruction set, every addressing
mode, the hardware multiply/divide registers, and precise per-opcode
cycle timing:

- ["Assembly for the SNES" by Ersanio](https://ersanio.gitbook.io/assembly-for-the-snes) —
  a complete, beginner-accessible book covering the full instruction set,
  addressing modes, and common idioms.
- [65816 opcode & cycle reference](https://wiki.superfamicom.org/65816-reference) —
  the authoritative per-instruction timing table.
